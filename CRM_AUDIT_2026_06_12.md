## THE BIG FIVE (in priority order)

### 1. Phone-format mismatch defeats ALL dedup → CRM fills with duplicates ★ foundational

Every modern writer (incl. the new `upsertLead`) normalizes to bare 10 digits and exact-matches
`findFirst({ where: { phone } })`. But 15/18 existing rows are `+919876543210`. `'9876543210' !==
'+919876543210'` → dedup misses → a duplicate is minted for every legacy contact who resubmits.
This is also why 5 dupes already exist. **Fix**: dedup on `phone: { endsWith: <10digit> }` in
`upsertLead.ts:79` (and aria/demo/catalog), and one-time normalize the 18 legacy rows to bare 10
digits. Until fixed, the CRM will fragment. Files: `src/lib/leads/upsertLead.ts`,
`src/app/api/aria/lead-capture/route.ts`, `demo/book`, `catalog/download`.

### 2. Eight lead writers are BROKEN (stale schema / invalid enums) → funnels lose leads, some roll back bookings

Verified against schema; they compile only because `prisma` is typed as a union with a mock client.

- `seo/leads/capture:120` → `assignedToId: 'system'` (no such user) → FK 500. CRM lead lost (route still 201s).
- `admin/students:158` → writes ~12 columns that don't exist on `leads`; missing required fields → "Add Student" always throws.
- `admission-application:83` → stale fields, missing `studentName/courseInterest/assignedToId/updatedAt` → every admission submission 500s (not wrapped).
- `leads/blog-query:15` → stale fields + free-text `source` → throws.
- `demo/book:348` → `determineLeadSource()` can return `GOOGLE_ADS`/`FACEBOOK_ADS` (NOT in LeadSource enum); lead create is in the SAME `$transaction` as the booking → **invalid enum rolls back the demo booking**. Paid/ad-click demos fail entirely; organic works.
- `leads/demo-booking:53` → `PAID_ADS`/`WEBSITE_FORM` (invalid enum) + missing `assignedToId`, in a transaction → LP demo funnel broken.
- `webhooks/leads:528` (create branch) → free-text source + missing `id`/`updatedAt` → external ad-platform lead webhooks never persist. (update branch is fine.)
- `catalog/download:85` → filters counselor on non-existent `isActive` column → throws; CRM lead skipped (non-blocking). Also invalid enum on its create.
  **Valid LeadSource values**: MANUAL_ENTRY, WALK_IN, PHONE_CALL, REFERRAL, WHATSAPP, EMAIL, SOCIAL_MEDIA, WEBSITE, ADVERTISEMENT, EVENT, OTHER. Map all ad sources → ADVERTISEMENT/SOCIAL_MEDIA.

### 3. The counselor Task / follow-up subsystem is DEAD → no "who to call next"

- `api/counselor/tasks:141` writes `status: 'TODO'` (enum is PENDING/IN_PROGRESS/COMPLETED/SNOOZED/CANCELLED); zod `type` allows `FOLLOW_UP`/`DEMO_REMINDER` (enum is FOLLOW_UP_CALL/DEMO_FOLLOWUP); `include: { users }`/`{ lead }` use non-existent relation names (`users_tasks_assignedToIdTousers`, `leads`); create omits required `updatedAt`. GET/POST/PATCH all throw.
- `followup_queue` action buttons send `action` in the JSON **body** but the route reads `searchParams.get('action')` → always null → 400 on every Execute/Cancel/Skip.
- `taskService.ts` automation writes `activities` with non-existent `type`/`performedBy` fields → throws.
- NOTE: the NEW `upsertLead` creates its task with the CORRECT enums, so auto-tasks land in the DB — but the counselor's task UI/API can't read or action them. So tasks exist but are unworkable.

### 4. No human is notified when a lead arrives (env keys unset)

All notification paths early-return / catch on missing keys → **admin gets nothing** today. The lead's own welcome WhatsApp also no-ops (gracefully). Only signal = logging into the CRM. Also `RESEND_API_KEY`/`SENDGRID_*`, `MSG91_AUTH_KEY`/`TWILIO_*` for email/SMS fallbacks.

### 5. Backfill tool is broken; crons mis-wired

- `admin/leads/migrate` reads `whatsappPhone`/`engagementScore`/`captureType`/`articleSlug` — none exist (field is `whatsappNumber`). Running it today creates **phoneless, non-deduped junk**. Don't run as-is. Safe backfill = a one-off script looping `content_leads` → `upsertLead({ phone: row.whatsappNumber, ... })` (once #1 is fixed).
- `vercel.json` crons for `followup-queue` + `followup-triggers` only export POST, but Vercel Cron issues GET → 405 → **followup_queue written but never processed** (whole `followupProcessor` engine dead). `nurturing`/`nurturing-sequence` correctly use GET. All need `CRON_SECRET`.
